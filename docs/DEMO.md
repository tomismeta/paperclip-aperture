# Demo

This is the fastest way to demo `paperclip-aperture` live inside Paperclip.

## Demo Thesis

Paperclip already knows about approvals, issues, comments, agents, and board actions.

Paperclip Aperture adds the live attention layer:

- what deserves attention now
- what should wait next
- what should remain ambient
- what the operator should actually do

The demo should prove that `Focus` is not another inbox. It is the ranked, operator-facing layer on top of Paperclip.

## What To Show

In 3-5 minutes, show:

1. `Focus` exists as a first-class destination inside Paperclip.
2. The `Now` card shows a recommended move, not just a raw issue title.
3. `Next` is a staged queue behind the current focus.
4. `Ambient` keeps resolved or low-pressure awareness visible without interrupting the operator.
5. The operator can act from Focus without leaving the plugin:
   - approve/reject
   - add a comment
   - acknowledge and clear the current state
6. The surface updates as new approvals, issue comments, and board decisions arrive.

## Suggested Live Flow

### 1. Start At The Sidebar

Open Paperclip and point out:

- `Focus` appears in the main navigation
- it sits beside Inbox
- it is not a separate app or runtime

Suggested line:

> Inbox tells you what changed. Focus tells you what deserves attention now.

### 2. Open Focus

Point out:

- `Focus` is the user-facing destination label
- `Powered by Aperture` is secondary
- the surface is organized into:
  - `now`
  - `next`
  - `ambient`

Suggested line:

> Paperclip remains the system of record. Aperture adds the live attention layer on top of it.

### 3. Explain The `Now` Card

Use the active frame.

Point out:

- the recommended move is the headline
- the Paperclip issue or approval sits underneath as supporting identity
- badges show the impact and driver
- the action rail lets the operator respond immediately

Suggested line:

> The top slot is not the latest record. It is the one thing the system thinks should own the operator right now.

### 4. Show Approval Then Re-stacking

Approve the active approval item and let the next issue-backed item promote into `Now`.

Point out:

- the surface re-ranks immediately
- `Next` moves into `Now`
- the queue behind it stays staged

Suggested line:

> This is the wedge. Focus is not mirroring the inbox. It is re-stacking the queue as the situation changes.

### 5. Show Inline Comment From Focus

Use the active issue-backed `Now` item and post a short operator note inline from the action rail.

Point out:

- the operator can respond without leaving Focus
- the comment lands on the underlying Paperclip issue
- the current item can remain in `Now` if the comment does not resolve the blocker

Suggested line:

> Focus is not just for triage. It can push work forward directly against the underlying Paperclip record.

### 6. Show Acknowledge And Clear

Acknowledge the active `Now` item and let the next queued item promote.

Point out:

- `Acknowledge` marks the current state as seen
- the current item clears from `Now`
- the next queued item steps up immediately

Suggested line:

> When the operator has seen or handled the current state, Focus clears it and advances the next most important item.

### 7. Show Semantic Routing

Use an issue-backed `Now` item.

Point out:

- the language is specific to the latest comment or request
- known agents are resolved dynamically when issue text refers to them
- stale blockers can fall out of `Now` when a resolving comment lands

Suggested line:

> The plugin is not just showing issue state. It is interpreting issue, approval, and agent signals into operator language.

### 8. Show Ambient

Move to `Ambient`.

Point out:

- the item is still visible
- it is quieter and non-interrupting
- it remains awareness, not action

Suggested line:

> Ambient is where the operator keeps context without giving it the keyboard.

## Backup Narrative

If the live data is quiet, explain the architecture with this one-liner:

> Paperclip emits host facts, the plugin interprets them into operator-ready Aperture events, Aperture ranks them, and Focus renders the result back inside Paperclip.

## What To Say If Asked

### "Did this require changing Aperture core?"

No. Aperture core was embedded as-is. The integration work lives in the plugin layer.

### "Did this require changing Paperclip core?"

No. The plugin uses Paperclip's plugin model and the published plugin SDK, including worker-side HTTP/config/telemetry surfaces where the host does not yet expose a first-class approval client.

### "What is the product boundary?"

- Paperclip = runtime, event source, host UI, host actions
- Aperture = continuity and attention mechanics substrate
- `paperclip-aperture` = semantic adapter + Paperclip-specific policy overlays + Focus UI
